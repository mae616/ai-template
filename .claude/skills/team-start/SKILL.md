# [チーム] エージェントチーム起動

## 入力: $ARGUMENTS
- タスクの概要（必須）
- 例: `ログイン機能の実装（フロント + API + テスト）`
- 例: `PR #45 の並列レビュー`

---

## 前提条件

### 環境
- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` が settings.json の env に設定済み
- tmux がインストール済み（`which tmux` で確認）
- Claude Code を `claude --teammate-mode tmux` で起動している

### 起動方法
```bash
# tmux分割モードでClaude Codeを起動
claude --teammate-mode tmux
```

> tmuxセッション外から起動すると、自動的にtmuxセッションが作られる。
> 既にtmuxセッション内なら、そのまま分割ペインが追加される。

---

## チーム構成（デフォルト: 開発チーム）

| ロール | 名前 | 絵文字 | 担当 | モデル |
|--------|------|--------|------|--------|
| リーダー | リーダーにゃんこ | 👑 | 調整・タスク分割・統合（デリゲートモード） | Opus |
| 開発者A | フロントエンドにゃんこ | 🎨 | フロントエンド実装 | Sonnet |
| 開発者B | バックエンドにゃんこ | ⚙️ | バックエンド/API実装 | Sonnet |
| 開発者C | インフラにゃんこ | 🏗️ | インフラ/テスト/CI | Sonnet |

---

## 実行手順

### 1. タスク分析とファイル分割

タスクを分析し、**各メンバーが別ファイルを担当する**ように分割する:

```markdown
## 📋 チーム作業計画

### タスク: {$ARGUMENTS}

### ファイル分割（コンフリクト防止）
| メンバー | 担当ファイル/ディレクトリ |
|----------|------------------------|
| 🎨 フロントエンドにゃんこ | src/components/, src/pages/ |
| ⚙️ バックエンドにゃんこ | src/api/, src/services/ |
| 🏗️ インフラにゃんこ | tests/, config/, CI設定 |

> ⚠️ 同じファイルを複数メンバーが編集しないこと！
```

> ⚠️ **ユーザー確認**: この分割で進めてよいか確認

---

### 2. チームメンバーをスポーン

以下のプロンプトでチームを作成する:

```
エージェントチームを作成してください。デリゲートモードで運用します。

チームメンバー3名をスポーンしてください:

1. 名前: "🎨 フロントエンドにゃんこ"
   担当: {フロントエンド担当の具体的タスク}
   担当ファイル: {具体的なファイルパス}
   モデル: Sonnet

2. 名前: "⚙️ バックエンドにゃんこ"
   担当: {バックエンド担当の具体的タスク}
   担当ファイル: {具体的なファイルパス}
   モデル: Sonnet

3. 名前: "🏗️ インフラにゃんこ"
   担当: {インフラ/テスト担当の具体的タスク}
   担当ファイル: {具体的なファイルパス}
   モデル: Sonnet

各メンバーへの共通指示:
- CLAUDE.md のペルソナ設定に従うこと（語尾「〜にゃ」）
- 担当ファイル以外は編集しないこと
- タスク完了後、完了報告をリーダーに送ること
- 不明点があればリーダーに質問すること
```

---

### 3. 進捗監視

リーダーは以下を監視する:
- 各メンバーの進捗状況（タスクリスト）
- ブロッカーの発生（メンバー間の依存関係）
- ファイルコンフリクトの兆候

---

### 4. クロスレビュー

全メンバーの実装完了後、クロスレビューを指示する:

```
全メンバーの実装が完了しました。クロスレビューを実施してください:

- 🎨 フロントエンドにゃんこ → ⚙️ バックエンドにゃんこのコードをレビュー
- ⚙️ バックエンドにゃんこ → 🏗️ インフラにゃんこのコードをレビュー
- 🏗️ インフラにゃんこ → 🎨 フロントエンドにゃんこのコードをレビュー

レビュー観点:
- 設計意図との整合性
- エッジケースの考慮
- セキュリティ（OWASP基本）
- テストカバレッジ
```

---

### 5. 統合とクリーンアップ

```markdown
## ✅ チーム作業完了

### 成果物
- {成果物リスト}

### クロスレビュー結果
- {指摘と対応のサマリー}

### 次のアクション
- [ ] コミット
- [ ] push
- [ ] チームクリーンアップ
```

チームのクリーンアップ:
```
チームメンバーをシャットダウンし、チームをクリーンアップしてください。
```

## 使い分けガイド

### チームモードを使う場面
- フロント + バック + テストにまたがる機能実装
- 複数モジュールの並列リファクタリング
- 複数視点でのPRレビュー・調査
- 仮説ベースのデバッグ（複数原因候補を並列検証）

### 通常モード（subagent）で十分な場面
- 単一ファイル/モジュールの変更
- 順序依存のある作業（A完了後にBを開始）
- 簡単な調査・ドキュメント作業
- トークンコストを抑えたい時

## 操作リファレンス

| 操作 | キー/コマンド |
|------|-------------|
| メンバー選択 | Shift+Up/Down |
| メンバーのセッション表示 | Enter |
| 現在のターン中断 | Escape |
| タスクリスト表示 | Ctrl+T |
| デリゲートモード切替 | Shift+Tab |
| tmuxペイン切替 | tmuxペインをクリック |

---

## 注意事項

- **トークンコスト**: チームモードは通常の3〜4倍以上のトークンを消費する
- **ファイルコンフリクト**: 同じファイルを複数メンバーが編集すると上書きが発生する
- **セッション再開不可**: in-processメンバーは `/resume` で復元されない
- **1セッション1チーム**: 同時に複数チームは作れない
- **ネストなし**: チームメンバーは自分のチームを作れない

## 連携スキル

- `/session-start` - セッション計画（チーム起動前に実行推奨）
- `/session-end` - セッション終了（チームクリーンアップ後に実行）
- `/basic-review` - 個別レビュー（チーム不要時）
- `/task-list` - タスク計画（チームのタスクリストと連携）
